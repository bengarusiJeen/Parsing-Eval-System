The Parser Evaluation System — How It All Works
The Cast (who runs where)
Your system is not one program. It is four separate things talking over HTTP, each on its own port:

Actor	Port	What it is	Who starts it
Frontend (Vite + React)	5173	The dashboard UI in the browser	npm run dev in frontend/
Eval Backend (FastAPI)	8000	Your system's API — the brain	uvicorn backend.app.main:app
Jeen parser-service	4004	External document parser (a separate project)	its own venv, in Jeen-Parsers/parser-service/
Azure OCR	cloud	Reads text from images in GT folders	nothing — it's a cloud API
The golden rule: the backend is the only one that talks to everyone. The frontend only knows port 8000. The parser only listens on 4004. They never talk directly.

The Story — One Evaluation, From Click to Result
Chapter 1 — The user picks documents
You open the browser at localhost:5173. The React app immediately calls GET http://localhost:8000/api/files.

On port 8000, files_routes.py wakes up, looks inside data/files_corpus/, and finds your 5 document folders (air defense_1, 5 עמודים, …). It sends back their names. The UI shows them as a list. You tick a few and choose a parser (base_text_parser). You click Run Evaluation.

Chapter 2 — The request arrives at the backend
The frontend sends POST http://localhost:8000/api/evaluate with a body like {"selected": ["air defense_1"], "parsers": ["base_text_parser"]}.

evaluation_routes.py catches it. It is a thin controller — it does no real work. It just hands the request to EvaluationService, which FastAPI handed it through dependencies.py.

Chapter 3 — The service prepares the run
evaluation_service.py does three things:

Wipes old reports — deletes the 4 old JSON files in reports/ so you never see stale data.
Builds a command — it does not run the evaluation itself. It builds a command line:

python -m backend.app.pipeline.main  data/files_corpus  --verbose  --output reports/general_report.json  --parser base_text_parser  --include "air defense_1"
Spawns a subprocess — it launches that command as a separate Python process and waits.
Why a subprocess? So a heavy, slow evaluation can't freeze the web server. The API stays responsive.

Chapter 4 — The pipeline runs (the real work)
The subprocess starts at pipeline/main.py. This is where the actual evaluation happens:

Step 1 — OCR pre-test. ocr.py scans every GT/ folder for imgX.png images. If a matching descX.txt is missing, it sends the image to Azure OCR (the cloud) and saves the text. (Your folders already have all desc files, so this step skips — but this is the one moment the system reaches out to Azure.)

Step 2 — Collect documents. utils.py walks data/files_corpus/ and keeps only folders that have both a GT/ subfolder and a document file.

Step 3 — For each document, evaluate it:

gt_loader.py reads the human-written GT/Text.txt, splits it into ==== blocks, and tokenizes it. This is the "correct answer".
parser.py → parser_backends.py takes the actual PDF/DOCX and sends it as POST http://localhost:4004/api/v1/parser/parse. This is the moment your backend talks to the Jeen parser on port 4004. Jeen reads the document and returns plain text. That text is saved to data/parsing_results/air defense_1.txt.
metrics.py compares the two: Coverage = how much of the GT the parser found; Noise = how much junk the parser added.
Step 4 — Pass 1 reports. reporting.py writes reports/general_report.json, and diagnostics.py classifies why things were missed (OCR splits, merged words, reversed word order…) into reports/diagnostics_report.json.

Step 5 — Pass 2 (postprocessing). postprocessing.py cleans up the parser's raw text (fixes punctuation, RTL Hebrew issues, rejoins split words), then the document is re-evaluated. This produces reports/postprocessing-general_report.json and reports/postprocessing-diagnostics_report.json. This is the "before vs after" comparison.

The subprocess now exits. Four JSON files sit in reports/.

Chapter 5 — The result travels back
EvaluationService sees the subprocess finished. It asks report_service.py to load all 4 JSON files, bundles them into one response, and returns it up the chain → route → frontend. The dashboard renders coverage, noise, and diagnostics.

Chapter 6 — Browsing afterwards (no re-run)
Once reports exist, the other endpoints just read them:

GET /api/results → returns the cached 4 reports.
GET /api/comparison → comparison_service.py + comparison_engine.py rank parsers against each other.
GET /api/stream_data → stream_service.py builds the colored side-by-side GT / raw / postprocessed view.
None of these touch the parser. They are fast because the slow work was already done and saved.

What You Need To Do To Run It
You need 3 terminals (the parser, the backend, the frontend), in this order:

Terminal 1 — Start the Jeen parser (port 4004)
This is a separate project with its own virtual environment.


cd "C:\Users\BenGarusi\Desktop\Jeen-Parsers\parser-service"
.venv\Scripts\python.exe -m uvicorn main:app --host 127.0.0.1 --port 4004
Wait until it's listening. Without this, every document fails to parse.

Terminal 2 — Start the Eval Backend (port 8000)

cd "C:\Users\BenGarusi\Desktop\Parsing-Eval-System"
python -m uvicorn backend.app.main:app --reload --port 8000
Check it: open http://localhost:8000/docs — you should see all 6 endpoints.

Terminal 3 — Start the Frontend (port 5173)

cd "C:\Users\BenGarusi\Desktop\Parsing-Eval-System\frontend"
npm install      # first time only
npm run dev
Open http://localhost:5173.

One-time prerequisites
backend/.env must contain AZURE_OCR_KEY and AZURE_OCR_ENDPOINT (it does). Only needed if GT folders have images without desc files.
The data/files_corpus/ folder must hold your document folders (it does — 5 of them).
The parser URL is read from backend/.env via PARSER_URL (defaults to http://localhost:4004/...).
Quick sanity check without the frontend

curl http://localhost:8000/api/health     # {"status":"ok"}
curl http://localhost:8000/api/files      # lists your 5 documents
